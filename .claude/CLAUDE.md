# Terraform Provider for Rubrik

## Build and Test

```bash
# Build
go build ./...

# Run unit tests
go test ./...

# Run acceptance tests (requires RSC credentials)
TF_ACC=1 go test -count=1 -timeout=120m -v ./...

# Lint and format
go vet ./...
gofmt -d .

# Regenerate documentation (NEVER manually edit docs/)
go generate ./...
```

## Project Structure

```
internal/provider/     # All resource, data source, and provider code
  names.go             # All field name constants (key* pattern)
  provider.go          # SDKv2 provider, resource/data source registration
  framework_provider.go  # Framework provider, framework resource registration
  resource_*.go        # SDKv2 resources
  data_source_*.go     # SDKv2 data sources
  framework_resource_*.go        # Framework resources (newer pattern)
  framework_data_source_*.go     # Framework data sources (newer pattern)
templates/             # Documentation templates (source of truth for docs)
  guides/              # Changelog, upgrade guides
  resources/           # Resource doc templates (optional per resource)
  data-sources/        # Data source doc templates (optional per resource)
docs/                  # AUTO-GENERATED by go generate - never edit directly
examples/              # Example .tf files referenced by generated docs
```

## Key Conventions

### Field Name Constants

All schema field keys are defined in `names.go` as constants. Never use string literals for field names in schema definitions.

```go
const keyMyField = "my_field"
```

### Two Provider Frameworks

The provider uses both SDKv2 and the Terraform Plugin Framework (muxed together). Newer resources use the Framework pattern (`framework_resource_*.go`), older ones use SDKv2 (`resource_*.go`). Follow the pattern matching the file you're working in.

### Description Helper

Use `description()` to wrap description constants. It converts acute accents to backticks since Go raw string literals can't contain backticks:

```go
const resourceExampleDescription = `
The ´rubrik_example´ resource manages examples in RSC.
`
```

### CRUD Pattern

- Start every CRUD function with `tflog.Trace(ctx, "functionName")`
- Access the RSC client via `m.(*client).polaris()` (SDKv2) or `r.client.polaris()` (Framework)
- Use SDK wrappers: `access.Wrap(client).SomeMethod(ctx, ...)`
- Handle `graphql.ErrNotFound` in Read (remove from state) and Delete (ignore)
- Always check errors from `d.Set()` calls

### Region Handling

Always use typed regions (`aws.Region`, `azure.Region`, `gcp.Region`) from the SDK, never raw strings.

### UUID Handling

Use the existing third-party UUID dependency for all UUID operations. Do not write custom UUID parsing or formatting.

## Documentation

**Never manually edit files in `docs/`** - they are generated by `go generate ./...`.

- Templates in `templates/` are the source of truth when they exist
- If no template exists, docs are generated from code descriptions and schema definitions
- Changelog: `templates/guides/changelog.md.tmpl`
- Upgrade guides: `templates/guides/upgrade_guide_v<VERSION>.md.tmpl`
- Do not use backticks inside markdown link text (e.g., use `[rubrik_foo docs](../resources/foo.md)` not `` [`rubrik_foo` docs](...) ``)

### Changelog Entry Format (ordered by priority)

1. Breaking changes: `* **Breaking Change:** [description]. [link to upgrade guide].`
2. Deprecations: `* **Deprecated:** [field] in [resource]. Use [replacement].`
3. New resources: `* New resource added for [name] which [description]. [[docs](link)]`
4. New data sources: `* New data source added for [name] which [description]. [[docs](link)]`
5. Features: `* Add support for [feature] in [resource]. [[docs](link)]`
6. Improvements: `* Improve [what] in [resource].`
7. Bug fixes: `* Fix a bug in [resource] where [description].`

## Testing

- Acceptance tests require `TF_ACC=1` and RSC credentials via `RUBRIK_SERVICEACCOUNT_FILE`
- Test function names use camelCase: `TestAccPolarisAwsAccount_basic`
- Prefer fakes over mocks for dependencies
- Cloud-specific tests require config files: `TEST_AWSACCOUNT_FILE`, `TEST_AZURESUBSCRIPTION_FILE`, `TEST_GCPPROJECT_FILE`, `TEST_RSCCONFIG_FILE`
- CDM (Data Center) tests are gated behind the `cdm` build tag and require `TEST_DATACENTER_FILE`. Passing `-tags=cdm` includes them alongside the default suite (it does not replace it). Run with `TF_ACC=1 go test -tags=cdm ./...`

## Go SDK Dependency

The provider depends on `github.com/rubrikinc/rubrik-polaris-sdk-for-go`.

```bash
go get github.com/rubrikinc/rubrik-polaris-sdk-for-go@<version>
go mod tidy
```

### Working with dependency source code

Use `go list` to find the filesystem location of dependencies:

```bash
go list -m -json github.com/rubrikinc/rubrik-polaris-sdk-for-go | jq -r .Dir
```

This respects `GOWORK` and local workspace configurations. Do not look in the module cache directly or use `go doc` to read source — always use `go list` to resolve paths.

### Important: never add replace directives

Never add `replace` directives to `go.mod`. If you need to work with a local copy of the SDK, use a `go.work` file instead.

## Release Process

### 1. Ensure SDK is on a semver tag

During development, commit-based pseudo-versions in `go.mod` are fine. Before releasing, the SDK must be pinned to a proper semver tag (e.g., `v1.4.0`). Tag the SDK first, then update the provider's `go.mod`.

### 2. Pre-release checks

```bash
go build ./...
go vet ./...
gofmt -d .
go generate ./...   # verify no files changed
go test ./...
```

### 3. Update changelog and upgrade guide

- Update `templates/guides/changelog.md.tmpl` with all changes since the last release
- If there are breaking changes, new features, or deprecations, create or update `templates/guides/upgrade_guide_v<VERSION>.md.tmpl`
- Run `go generate ./...` to regenerate docs
- Commit, push, and merge via PR

### 4. Tag and release

```bash
git checkout main && git pull origin main
git tag v<MAJOR>.<MINOR>.<PATCH>
git push --dry-run --tags   # verify first
git push --tags              # triggers GitHub Actions release
```

GitHub Actions runs GoReleaser which builds binaries for all platforms, signs artifacts with GPG, creates a GitHub release, and publishes to the Terraform Registry.

### Version numbering

- **Major** (v2.0.0): major refactors or large breaking changes
- **Minor** (v1.7.0): new features, may include small breaking changes with an upgrade guide
- **Patch** (v1.7.1): bug fixes, backward compatible
